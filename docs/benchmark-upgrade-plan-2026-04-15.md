# MYmd 对标升级计划（2026-04-15）

## 对标对象

- `doocs/md`
- `Cherry Markdown`
- `UEditor`

## 结论摘要

MYmd 当前已经有桌面本地编辑、工作区、知识索引、双栏编辑、KaTeX、Mermaid、AI 助手、主题与排版设置这些基础能力，但和对标项目相比，仍然有三类明显短板：

1. 工作区与多文档生产力不够完整
2. 编辑器高级交互与工具化能力不够深
3. AI、导出、插件化和跨窗口协作还没有形成产品级闭环

## 对标能力差距

### 相比 `doocs/md`

`doocs/md` 的公开说明强调了主题样式、内容管理、多图床、AI 助手，以及私有化部署方式（CLI / Docker）和批量导入导出、更多主题、MCP Server 等持续需求。  
这说明它的优势不只在编辑器，而在“内容生产 + 分发 + 发布”的一体化体验。

我们当前还没做好的点：

- 图片资源工作流不完整
  - 缺少图床/资源托管策略
  - 缺少粘贴图片后的统一落盘、重命名、迁移与链接修复
- 导出链路不够强
  - 已有 HTML 导出，但缺少 PDF / 图片 / Docx 等更强导出矩阵
- AI 助手更偏“文档内建议”，还不是任务型工作流助手
  - 缺少模板化 prompt、上下文选择、历史版本比对、批量处理
- 缺少面向自动化/外部接入的能力
  - 没有命令行入口
  - 没有可被外部工具调用的本地服务或 MCP 入口

### 相比 `Cherry Markdown`

`Cherry Markdown` 强项很清晰：支持富 Markdown 语法、富文本粘贴转 Markdown、表格编辑、表转图、音视频、浮动工具栏、选择气泡工具栏、快捷键、自定义工具栏、自动补全、流式 AI 场景、局部渲染/局部更新，以及多实例。  
这些能力说明它在“编辑过程效率”和“嵌入式扩展性”上成熟度更高。

我们当前还没做好的点：

- 缺少富文本粘贴转 Markdown 的高保真链路
- 缺少表格增强能力
  - 可视化增删行列
  - 表格格式化
  - 表转图
- 缺少上下文工具条
  - 选中文本后的气泡工具栏
  - 新行/空块触发的轻量快捷操作
- 缺少自动补全与输入建议
  - 标题、链接、wiki-link、标签、文件名建议
- 缺少多实例 / 多窗口协作能力
  - 当前无法像 Word 那样把标签拖出为独立窗口
- 缺少流式 AI 与编辑态深度联动
  - 还没有“边生成边落文档结构”的体验

### 相比 `UEditor`

`UEditor` 是典型富文本编辑器路线，虽然仓库已归档，但它在工具栏、插件扩展、自动保存、纯文本粘贴、搜索替换、附件/图片/视频等编辑器基础设施上仍然有参考价值。

我们当前还没做好的点：

- 工具栏可配置性不够高
- 插件化边界仍偏弱
  - 还没有稳定的第三方扩展模型
- 编辑器命令体系还不够完整
  - 粘贴策略
  - 自动排版
  - 批量格式化
  - 更成熟的查找替换反馈
- 多媒体与附件工作流还不完整

## 这轮已落地

### 1. 工作区嵌套 Markdown 识别修复

- 修复 wiki-link 在嵌套目录中的相对路径解析
- 增加对 `./`、`../` 和显式工作区根路径 `/...` 的支持
- 统一工作区递归收集 Markdown 文件的路径工具
- 为嵌套目录链接解析补了测试

### 2. 标签页双击重命名

- 支持双击标签页进入重命名
- 已保存文件会真正改名
- 会同步打开中的 tab 路径与标题
- 会批量重写工作区内受影响的 wiki-link
- 未保存标签页则只修改标签标题

## 下一阶段升级优先级

### P0：稳定性与多文档闭环

1. 完成工作区索引一致性治理
2. 修复文件树、反向链接、知识图谱、全局搜索对嵌套路径的统一认知
3. 补文件移动、目录重命名、冲突命名、重复文件名场景测试
4. 给重命名、移动、删除建立统一事务反馈

### P1：生产力体验补齐

1. 标签页上下文菜单
   - 重命名
   - 复制路径
   - 在文件树定位
   - 在新窗口打开
2. 多窗口 MVP
   - 先支持“在新窗口打开当前文档”
   - 再做真实拖拽脱离窗口
3. 搜索替换增强
   - 匹配数
   - 替换预览
   - 作用范围（当前文档 / 当前文件夹 / 全工作区）
4. wiki-link / 标签 / 文件名自动补全

### P2：编辑器高级能力

1. 选区气泡工具栏
2. 新段落快捷插入条
3. 表格增强
   - 增删行列
   - 对齐同步
   - 格式化
   - 表转图
4. 富文本粘贴转 Markdown
5. 图片粘贴与资源管理闭环

### P3：AI 与导出工作流

1. AI 助手增加任务模板与上下文源选择
2. 流式生成直接回填到结构化文档
3. 导出扩展到 PDF / 图片 / Docx
4. 增加本地自动化入口
   - CLI
   - 本地服务
   - MCP

## 来自对标项目的 issue / 反馈启发

### `doocs/md`

从公开 issue 列表可以看到，用户持续关注这些方向：

- 文件保存在服务器
- 批量导入导出 md 文件
- 跨平台移动端 / 桌面端支持
- CLI 工具
- MCP Server 适配
- 主题数量继续增加
- 所见即所得和发布结果不一致

映射到 MYmd 的动作：

- 增加工作区批量导入导出
- 统一“编辑态 vs 导出态”渲染一致性验证
- 预留自动化接口而不是只做 UI 面板

### `Cherry Markdown`

从公开 issue 与能力说明能看到这些高频诉求：

- 复制 HTML 后换行异常
- 纯预览状态误进入编辑态
- 导出 Docx
- 编辑器字号变化导致图标重叠
- 目录点击/路由联动
- 自定义 LLM 配置

映射到 MYmd 的动作：

- 增加粘贴归一化测试
- 强化预览态与编辑态边界
- 做编辑器缩放/字体变化下的工具栏回归测试
- 做导出格式扩展
- AI 接入支持多供应商模板化配置

### `UEditor`

仓库已归档，README 直接说明 issue 太多且维护时间不足、暂时关闭 issue。  
这对我们的启发不是“抄功能”，而是要避免老式富文本编辑器常见的维护陷阱：

- 工具栏功能越来越多，但测试和边界条件跟不上
- 插件扩展点多，但约束不清晰
- 粘贴、自动保存、媒体上传这类高频能力如果缺少统一抽象，会长期变脆

映射到 MYmd 的动作：

- 在增加编辑能力前先补命令层和插件边界
- 每个高频输入链路必须有回归测试
- 多媒体、粘贴、自动保存、导出不要分散在多个组件各自处理

## 建议的执行顺序

1. 多窗口 MVP 设计与启动参数方案
2. 工作区文件移动/目录重命名/重复名冲突的回归测试
3. Tab 上下文菜单 + 在文件树定位 + 新窗口打开
4. 搜索替换增强
5. wiki-link 自动补全
6. 富文本粘贴与图片资源工作流
7. 表格增强与导出增强
